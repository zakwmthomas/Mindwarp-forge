## Rollback gap and alternatives

The current Kernel candidate lifecycle is `Proposed -> Approved -> Promoted`,
with terminal `Rejected` or `Quarantined` paths. `reject_candidate` accepts only
a `Proposed` candidate. There is no event or state for withdrawing an approved
candidate or superseding a promoted canonical baseline.

| Alternative | Finding |
|---|---|
| Leave the old candidate `Promoted` and promote a correction | Rejected: two apparently canonical baselines remain active and the old status is misleading. |
| Treat a control-plane rollback record as candidate rollback | Rejected: that record is authority-negative and applies to failed/blocked work gates, not Kernel candidate state. |
| Change documentation only | Rejected: generated or authored projections cannot override the protected ledger. |
| Append an explicit owner-authorized candidate-supersession event and state | Selected for design. It preserves history, makes the prior baseline inactive, binds correction evidence and an optional replacement, and has no application/runtime effect. |

The pre-H7 rollback target is **no promoted humanoid proof baseline**, while the
verified H6 manifest remains immutable evidence. After any later replacement,
the rollback/supersession record must identify both old and new candidate IDs
and retain the old evidence rather than deleting it.

## Cheapest proof before Kernel work

Before any protected-Kernel change, a capability-free typed fixture must prove:

1. the narrow candidate package binds the exact H6/H5 IDs, claims, non-claims,
   required two-step owner actions, and rollback target;
2. production-asset, merged-wire/visual, missing-H5, stale-H6, overclaim,
   ambiguous-name, auto-authority, and no-rollback packages fail closed;
3. a lifecycle model rejects supersession of proposed/rejected candidates,
   forged actors, missing correction evidence, self-replacement, and stale IDs;
4. promoted-to-superseded history replays deterministically and retains the
   old evidence; and
5. supersession has no import, execution, application, publishing, or runtime
   effect.

Only after that cheaper proof passes should a separately authorized Kernel
change be proposed. No owner approval or promotion phrase should be requested
until the lifecycle gap is closed and the complete Forge gate passes.

## Protected-Kernel implementation authorization gate

The smallest justified protected change is now bounded:

1. add `CandidateState::Superseded` and `EventType::CandidateSuperseded`;
2. add an append-only `supersede_candidate` transition for `Approved` or
   `Promoted` candidates only;
3. require `DirectProjectUser`, `ExplicitUserAuthorization`, exact candidate
   binding, retained lowercase-SHA-256 correction evidence, and optional
   non-self replacement candidate binding;
4. preserve the old evidence and event history on replay and SQLite reopen;
5. extend the explicit-authorization command/UI with an exact supersession
   phrase while retaining separate approval and promotion actions;
6. reject proposed/rejected/quarantined/superseded sources, forged/imported
   actors, stale IDs, missing evidence, self-replacement, repeated action, and
   unknown versions; and
7. prove no code/asset application, import, execution, publishing, runtime
   selection, or deletion effect before the complete Forge gate.

This change would make promotion reversible at the authority ledger without
promoting the H7 humanoid package itself. It modifies the protected Kernel and
therefore requires separate owner authorization under the Working Covenant.
