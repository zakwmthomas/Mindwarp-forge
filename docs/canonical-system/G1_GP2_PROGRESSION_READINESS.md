# G1 GP2 progression readiness

Status: owner-authorized design receipt accepted and progression semantics
frozen for red tests; source implementation has not begun.

## Dependency and authority

GP1 is structurally complete through one engine-neutral deterministic
six-phase loop across all five fixed sessions, repairable failure, typed stable
stops, strict run/history receipts, exact optional C3A context, later-state
consequence, independent acceptance, and the complete registered Forge gate.
That proof remains `prototype_tested` and grants no runtime authority.

GP2 may design and later prove five distinct typed progression lanes:
knowledge, access, relationship, construction, and power. It must enumerate
explicit sources, sinks, reset boundaries, conversions, and exploits; preserve
named decisions; and use deterministic strategy simulations. It may not add a
universal currency, positive conversion cycle, dominant conversion, grind,
runtime, graphics, persistence, network, monetization, Greenfield dependency,
C3B claim, GP3 grammar, or GP4 vertical.

## Selected design

The accepted read-only design receipt freezes:

- `ProgressionLedgerV1`, which binds its source base-loop-ledger digest and
  processed progression receipts;
- `apply_progression(session, replay-validated terminal state, fixed rule
  registry, prior ledger)` and explicit per-session/outcome rules only;
- distinct knowledge, access, relationship-event, construction, and capability
  records exactly as specified by `G1_GP2_PROGRESSION_DESIGN.md`;
- receipt identity across run, event, session, outcome, terminal digest, rule,
  emitted record IDs, world transitions, and named decision;
- nonrecursive guarded conversions, typed liabilities, anti-farming causal
  preconditions, and no same-event or same-subject cycles;
- deterministic steward-builder, urgency-discovery, and cautious-mapper
  strategies with a pairwise-incomparable viable-set proof;
- the exact S1 capability allowlist and no capability for retreat.

## Frozen input-authority boundary

Durable GP2 progression may consume only a replay-validated terminal
`BaseLoopStateV1` and must bind that state's canonical digest. Progression rules
must be explicit per session and outcome. GP1 opportunity-cost strings and
arbitrary `PreparationV1.tool_id` values are caller-supplied and cannot create
durable progression. Failure costs remain attempt-local unless a future
fixture-owned rule validates them. A power or capability grant requires both an
exact allowlisted prepared tool and a successful allowlisted outcome; combat
diversion and free-form tool text grant nothing. This authority boundary is
frozen now, while the actual lane records and progression rules remain pending
the design receipt.

## Provisional adversarial matrix

Before implementation, the final design must be falsifiable against:

- knowledge spent as currency, access transferred as fungible value,
  relationships reduced to a scalar meter, construction reduced to inventory,
  or power silently authorizing social or epistemic outcomes;
- any source without a named consequence, sink without a decision, reset that
  duplicates value, conversion with non-negative round-trip gain, or path that
  dominates every strategy;
- a single resource unlocking all five lanes, repeated low-risk activity
  replacing authored choices, combat completing progression, or retreat
  erasing remembered state;
- fabricated state, noncanonical bytes, reordered history, duplicate run
  receipt, foreign session consequence, or stale/forged C3A context;
- filesystem, engine, UI, database, network, Greenfield, C3B, GP3, or GP4
  imports and authority claims.

## Readiness decision

The design is bounded, engine-neutral, reversible, and falsifiable. GP2 red
tests may now be frozen from this record and the accepted design receipt.
Source implementation remains paused until this readiness result is reported;
GP3 and all runtime work remain forbidden.
