# G1 Product Portfolio and C3 Dependency Rebaseline

**Status:** owner-authorized program adjudication; dependency and product-boundary correction, not runtime or release authority.

## Canonical product model

`Forge Core -> proven reusable capabilities -> domain creative suites -> isolated product shells`

- Forge Core owns evidence, provenance, requirements, planning, replay, validation, metrics and agent coordination.
- Proven reusable capabilities are deterministic, versioned contracts promoted only after two independent real consumers prove the common seam.
- Domain creative suites are assembled when products demand them, not as speculative platform inventory.
- Isolated product shells own their UX, persistence, identity, credentials, billing and release lifecycle.

Development is product-driven and **vertical-then-extract**. Mind Warp is the first demanding customer. Product code is not generalized from prose, one disposable oracle or one consumer.

## Product boundaries

- Mind Warp game is the first delivery priority and playable-vertical consumer.
- Mind Warp Companion follows the first playable game.
- Yard Atlas remains an independent product, independently usable and releasable on the Play Store.
- Forge may index Yard Atlas evidence and later host a versioned garden-domain compatibility contract, but there is **no repository, database, UI, authentication, billing, or release authority merge**.
- Greenfield physical-garden geometry and coefficients cannot satisfy Mind Warp physical evidence.

## C3 dependency correction

- **C3A - dependency-sufficient causal-world foundation:** promoted. Its exact seam is a validated `WorldGenerationInput` and replayed `CausalWorldPacket` v1 with nested identity and provenance binding. `addressable-world-binding` imports only those records and `validate_world_packet`.
- **C3B - physical and presentation fidelity:** evidence-blocked. Project-specific physical scale, calibrated coefficients, applicability, received energy, visibility and presentation fidelity remain unavailable or unproven.
- **C3 umbrella:** superseded by this split. This does not claim full C3 closure.

C4 may depend on C3A because its current proof consumes exactly that seam. C3B does not block the first playable vertical; any feature needing a C3B fact must still fail closed or use an explicitly nonphysical authored fixture.

## Gameplay route

GP0 selects the player promise, non-goals and five representative sessions. GP1 proves a fixed-content base loop. GP2 proves progression, knowledge and economy choices. GP3 proves encounter grammar including noncombat paths. GP4 freezes one runtime-independent vertical contract.

Runtime selection remains behind the existing G1/R1 owner gate. No engine, network model, monetization model, historical-code reuse or C3B closure is granted here.
