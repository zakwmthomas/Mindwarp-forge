# Worker Governance System

The Forge worker is governed by a versioned operating system, not one prompt.
It applies universally to every Forge package and evolves only through recorded,
verified improvements.

## Layers

1. **Policy registry** — non-negotiable authority, evidence, macro-to-micro,
   recording, and ROI rules.
2. **Worker prompt specification** — concise executable behavior derived from
   policy; never the sole source of governance.
3. **Planning system** — Master Closure Register, dependency map, work package,
   and Batch State select/sequence actual work.
4. **Learning ledger** — observations, failure modes, improvements, expected
   benefit, and regression signals.
5. **Verification system** — tests and gates decide whether a technical or
   workflow change is retained.
6. **Stage-quality gates** - macro/micro context freshness, rendered-asset
   fitness, and simulation-first escalation are mandatory inputs to material
   work rather than optional worker judgement.

## Self-governing loop

`observe friction -> classify macro/group/contract cause -> propose change ->
apply bounded change -> verify -> record outcome -> promote/revise/revert`

At every three completed meaningful batches, audit all five layers. If a prompt
change alone cannot prevent recurrence, update the planning record, policy,
template, contract, or test guard instead. A learned rule becomes universal
only after it is recorded in the policy registry with rationale and a
regression signal.

## Boundary

The system may improve its workflow and reversible local tooling. It may not
grant itself authority, change owner decisions, spend, publish, use credentials,
weaken security, choose an engine, or promote generated code.
